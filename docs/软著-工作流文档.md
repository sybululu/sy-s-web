## 1. 系统架构总览

### 1.1 软件概述

PrivacyGuard 是一款基于人工智能的隐私政策合规审查 SaaS 平台，面向企业法务团队、数据合规官及开发者，提供**自动化的隐私政策违规检测、法律依据检索、智能整改建议生成**三大核心能力。

系统采用 **「单项目双模式」** 架构设计：
- **未登录态**：展示 Hero 营销站（Landing + Pricing），用于产品展示和用户转化
- **已登录态**：展示 B 端产品界面（Dashboard 审查工作台），提供完整的合规审查功能

## 2. 前端工作流详解

### 2.1 应用启动流程

```
浏览器访问 → index.html 加载
    ↓
main.tsx 执行
    ├── ReactDOM.createRoot('#app')
    ├── <BrowserRouter> 包裹（关键！解决 Hero 内部 <Link> useContext null 崩溃）
    └── <App /> 渲染
        ↓
App.tsx 渲染逻辑:
    ├── AuthContext.Provider 包裹全局
    ├── <Routes> 路由匹配:
    │   ├── path="/" → <Landing /> (未登录时显示营销站)
    │   ├── path="/pricing" → <Pricing />
    │   ├── path="/login" → <Login /> (全屏登录页)
    │   └── 未匹配路径 → <Navigate to="/" />
    ├── <Navbar /> 固定顶部导航
    └── <Footer /> 底部栏
```

**关键技术决策说明**：

为什么必须在 `main.tsx` 用 `<BrowserRouter>` 包裹 `<App />`？
- `Hero.tsx` 和 `Landing.tsx` 内部使用了 React Router 的 `<Link>` 组件
- `<Link>` 通过 `useContext(RouterContext)` 获取路由上下文
- 如果 BrowserRouter 放在 `App.tsx` 内部且条件渲染（如登录后才渲染），会导致 Hero 组件在渲染时找不到 RouterContext，触发 useContext return null 崩溃
- 因此将 BrowserRouter 提升到 `main.tsx` 最外层，确保所有子组件都能访问路由上下文

### 2.2 用户认证工作流

#### 2.2.1 登录流程

```
用户打开 /login 页面
    ↓
<Login /> 组件挂载
    ├── useState: email, password, isRegister, isLoading, ...
    ├── useState: showPolicyModal (协议弹窗状态)
    └── useEffect: 检查已登录状态 → 已登录则 redirect('/')

用户填写邮箱密码 → 点击"登录"
    ↓
handleLogin() 异步函数执行:
    ├── 表单验证 (email 格式、password 长度)
    ├── setIsLoading(true)
    ├── POST /api/v1/auth/login { email, password }
    │   ├── 成功: { token, user: { id, email, name } }
    │   │   ├── login(token, user) 写入 AuthContext
    │   │   ├── localStorage.setItem('token', token)
    │   │   └── navigate('/') → 回到首页（此时已登录，显示 Dashboard）
    │   └── 失败: 显示错误提示 toast
    └── setIsLoading(false)
```

#### 2.2.2 注册流程

```
点击"没有账号？立即注册" → isRegister = true
    ↓
表单切换为注册模式（额外 name 字段）
    ↓
填写姓名/邮箱/密码 → 点击"创建账号"
    ↓
handleRegister() 执行:
    ├── 先 setShowPolicyModal(true) → 弹出《服务条款》弹窗
    │   └── PolicyModal 组件:
    │       ├── 用户必须滚动到底部（进度条 100%）
    │       ├── "我已阅读并同意"按钮从 disabled 变为可点击
    │       └── onAccept() → 触发实际注册
    ├── POST /api/v1/auth/register { email, password, name }
    │   ├── 成功: 自动切换回登录模式 + toast 提示
    │   └── 失败: 显示错误（如"邮箱已注册"）
    └── 关闭弹窗
```

#### 2.2.3 AuthContext 状态管理

```typescript
// contexts/AuthContext.tsx 核心逻辑
interface AuthContextType {
  user: User | null;          // 当前登录用户信息
  token: string | null;        // JWT token
  login: (token: string, user: User) => void;   // 登录动作
  logout: () => void;                           // 登出动作
  isAuthenticated: boolean;                     // 是否已认证
}

// token 存储策略: localStorage（持久化刷新不丢失）
// 请求拦截器: axios instance 自动注入 Authorization: Bearer <token>
// 响应拦截器: 401 状态码自动清除 token 并跳转 /login
```

### 3.2 核心审查管线（/api/v1/analyze）

这是系统最核心的 AI 处理流程，采用**两阶段三重判断**架构。

```
POST /api/v1/analyze
Body: { text: "隐私政策全文...", source_type: "text" }
Headers: Authorization: Bearer <jwt_token>
    ↓
═══════════════════════════════════════════════
  预处理阶段
═══════════════════════════════════════════════

Step 1: 文本切分
    split_into_sentences(text)
    └── re.split(r'[。；\n]+', text) → List[sentence]
    └── 过滤: len(strip) > 5 的句子保留
    ↓
    例: "我们收集您的位置信息、设备信息、联系人信息等。"
      → ["我们收集您的位置信息、设备信息、联系人信息等"]

═══════════════════════════════════════════════
  第一阶段: 二分类前置过滤（三重判断）
═══════════════════════════════════════════════

对每条 sentence 依次执行:

  ┌──────────────────────────────────────────┐
  │  第 0 关: 关键词熔断（最高优先级）         │
  │  keyword_circuit_breaker(sentence)        │
  │                                          │
  │  扫描 CIRCUIT_BREAKER_KEYWORDS 词表:     │
  │  - "默认勾选"/"视为同意"/"即表示同意"     │
  │  - "生物识别"/"指纹"/"人脸识别"           │
  │  - "永久保存"/"无限期"                   │
  │  - 共 30+ 个流氓表述关键词                │
  │                                          │
  │  命中? → 直接 ⚠️ 强制违规 (跳过所有模型)  │
  │  未命中? → 进入第 1 关                    │
  └──────────────────────┬───────────────────┘
                       │ 未命中
                       ▼
  ┌──────────────────────────────────────────┐
  │  第 1 关: 二分类模型推理                   │
  │  CustomBertBinaryModel(sentence)          │
  │                                          │
  │  RoBERTa → fc(768→2) → softmax           │
  │  输出: safe_prob / risk_prob              │
  │                                          │
  │  risk_prob > 0.35 (严格阈值)?            │
  │    → ⚠️ 高置信违规                        │
  │                                          │
  │  risk_prob < 0.15 (安全下界)?            │
  │    → ✅ 高置信安全                        │
  │                                          │
  │  0.15 ≤ risk_prob ≤ 0.35 (模糊地带)?    │
  │    → ⏳ 标记需 Phi-4 二审                 │
  └──────────────────────┬───────────────────┘
                       │ 模糊地带
                       ▼
  ┌──────────────────────────────────────────┐
  │  第 2 关: Phi-4 LLM 二审法官              │
  │  llm_judge_violation(sentence)            │
  │                                          │
  │  Prompt: "请严格判断以下句子是否包含违规   │
  │          逻辑。只回答'是'或'否'"          │
  │  Model: Phi-4 Mini (max_tokens=8)        │
  │                                          │
  │  回答含"是"? → ⚠️ 违规 → 进入第二阶段    │
  │  回答含"否"? → ✅ 安全 → 跳过             │
  │  LLM 调用失败? → 保守策略: ⚠️ 违规        │
  └──────────────────────────────────────────┘

═══════════════════════════════════════════════
  第二阶段: 11 类多分类 → 12 类违规映射
═══════════════════════════════════════════════

仅对通过第一阶段（判定为违规）的句子执行:

  roberta_predict(sentence):
    ├── tokenizer_roberta(sentence, max_length=150)
    ├── model_roberta(**inputs) → logits tensor
    ├── sigmoid(logits) → 11 类概率 [p0, p1, ..., p10]
    ├── max_idx = argmax(probs) → 最高概率类别索引
    └── map_to_12_classes(probs, confidence)
        │
        │  映射关系 (mapper.py):
        │  0(数据收集) → ["I1", "I4"]
        │  1(权限获取) → ["I3"]
        │  2(共享转让) → ["I5", "I6", "I7"]
        │  3(使用) → ["I2"]
        │  4(存储方式) → ["I8"]
        │  5(安全措施/销毁) → ["I9"]
        │  6(特殊人群) → ["I3"]
        │  7(权限管理) → ["I10", "I12"]
        │  8(联系方式) → ["I11"]
        │  9(政策变更) → ["I11"]
        │  10(停止运营) → ["I9"]
        │
        └── → List[violation_id] 如 ["I1", "I4"]

═══════════════════════════════════════════════
  后处理: 评分计算 + 结果组装
═══════════════════════════════════════════════

对每个 detected violation:
    ├── RAG 检索法律依据:
    │   get_legal_basis_from_riolation_id(violation_id, context=sentence)
    │   ├── retriever.retrieve_by_violation_type(violation_id, context)
    │   │   ├── query = f"{sentence} {violation_keywords}"
    │   │   ├── vector_store.search_by_violation(query)
    │   │   │   └── FAISS 语义搜索 (余弦相似度, top_k=5)
    │   │   └── → List[SearchResult] (法律条文+相关度)
    │   └── format_legal_paired() → 引用+正文配对格式
    │
    └── 组装 violation item:
        {
          indicator: "过度收集敏感数据",
          violation_id: "I1",
          snippet: "原始违规句子",
          legal_basis: "《个人信息保护法》第六条\n正文...",
          confidence: 0.82,
          triggered_by: "model_high",  // 触发来源标记
        }

评分计算:
    violation_flags = { "I1": 1, "I3": 1, ... }  // 每种违规只扣一次
    penalty = Σ(INDICATORS[v_id].weight * flag)    // 权重求和
    total_score = max(0, 100 - penalty * 100)     // 百分制

风险等级判定:
    score >= 70 → "低风险"
    40 <= score < 70 → "中等风险"
    score < 40  → "高风险"

持久化:
    Project(
        id=f"p{uuid12}",
        user_id=current_user.id,
        name="审查-{timestamp}",
        score=total_score,
        risk_level=risk_level,
        result_json=json.dumps(violations_list),
        raw_text=text[:5000]
    )
    db.add(project) → db.commit()

返回 AnalyzeResponse:
{
    id, name, score, risk_level,
    violations: [...],           // 违规列表
    sentence_results: [...]      // 逐句明细（测试用）
}
```

```
POST /api/v1/rectify
Body: {

    ↓
═══════════════════════════════════════════════

═══════════════════════════════════════════════

get_legal_basis_from_rag(violation_type, context=original_snippet)

═════════════════════════════════════════════════

【mode = "summary"】摘要解读模式:
    System: "你是隐私合规顾问...绝对不要大段引用法律条文原文"

        一、问题诊断（违反哪条法律的具体要求）

    User: 结构化改写指令:
        【原条款】{original_snippet}
        【法律依据 — 阅读后理解，不可出现在输出中】{legal_content}
        【本次改写必须执行的操作】{EDIT_INSTRUCTIONS[violation_type]}

    EDIT_INSTRUCTIONS 针对 12 类违规各有专门指令:

═══════════════════════════════════════════════

════════════════════════════════════════════════

    │       └── 不可重试 (401/400) → 立即 raise
    └── 全部耗尽 → raise LLMCallError

═══════════════════════════════════════════════
  Step 4: 返回结果
════════════════════════════════════════════════
```

### 4.3 LLM 调用稳定性架构

```
call_llm_with_retry() 统一包装器:
    │
    ├── Circuit Breaker (熔断器)
    │   ├── 连续失败 ≥ 5 次 → 熔断 60s
    │   └── 熔断期间直接跳过调用，快速降级
    │
    ├── Retry with Exponential Backoff (指数退避重试)
    │   ├── base_delay = 1.0s (二审 0.5s)
    │   ├── delay × 2^attempt (1s → 2s → 4s)
    │   └── max_retries = 3 (二审 2 次)
    │
    ├── Error Classification (错误分类)
    │   ├── 可重试: 429(限速), 5xx(服务错), timeout, connection
    │   └── 不可重试: 401/403(认证), 400(无效请求), SSL 错
    │
    └── Non-blocking (非阻塞)
        └── asyncio.to_thread() → 不阻塞 FastAPI 事件循环
```

### 4.4 11 类 → 12 类映射详解

```
RoBERTa 模型训练时输出 11 类（CAPP-130 数据集类别），
但论文定义了 12 类违规指标体系。
Mapper 负责将 11 类映射到 12 类（多标签映射，一个源可对应多个目标）。

┌────┬──────────────┬─────────────────────────┬──────────┐
│Idx │ 11类原始名称  │ 映射目标(12类)          │ 映射数量  │
├────┼──────────────┼─────────────────────────┼──────────┤
│ 0  │ 数据收集      │ I1(过度收集) + I4(范围超出)│   2      │
│ 1  │ 权限获取      │ I3(未获明示同意)          │   1      │
│ 2  │ 共享转让      │ I5(第三方范围)            │          │
│    │              │ + I6(单独授权) + I7(用途)  │   3      │
│ 3  │ 使用目的      │ I2(未说明目的)            │   1      │
│ 4  │ 存储方式      │ I8(留存期限)             │   1      │
│ 5  │ 安全措施/销毁 │ I9(销毁机制)             │   1      │
│ 6  │ 特殊人群      │ I3(未获明示同意)          │   1      │
│ 7  │ 权限管理      │ I10(用户权利)+I12(响应时限)│  2      │
│ 8  │ 联系方式      │ I11(行使途径)            │   1      │
│ 9  │ 政策变更      │ I11(行使途径)            │   1      │
│ 10 │ 停止运营      │ I9(销毁机制)             │   1      │
└────┴──────────────┴─────────────────────────┴──────────┘

12 类全覆盖检查: I1✅ I2✅ I3✅ I4✅ I5✅ I6✅ I7✅ I8✅ I9✅ I10✅ I11✅ I12✅
```

### 8.3 RAG 检索流程

```
查询: violation_type="I1" + context="收集位置信息..."
    ↓
Retriever.retrieve_by_violation_type("I1", context)
    ↓
构造查询语句:
    query = "{context} 过度收集 敏感数据 最小必要 超范围收集"
    ↓
VectorStore.search_by_violation(query, top_k=5)
    ↓
FAISS 语义检索:
    1. embedding_model.encode(query) → query_vector (384维)
    2. faiss.normalize_L2(query_vector)
    3. index.search(query_vector, top_k*2)  // 多搜一些防过滤
    4. 余弦相似度排序 → top_k 结果
    ↓
后处理:
    1. 去重 (article_id level)
    2. Article → SearchResult 转换
    3. 补充 law_name (PIPL→个人信息保护法)
    4. 生成 law_reference 字段
    ↓
返回: List[SearchResult]
```

| 参数 | 值 | 说明 |

| 相似度度量 | 余弦相似度 (IndexFlatIP + L2 归一化) | |
| 索引类型 | FAISS IndexIDMap (IndexFlatIP) | |
| 分块策略 | 按法律条文整条存储（不分 chunk） | |

| 校验点 | 规则 | 实现 |
|--------|------|------|
| 文本长度 | 10 ≤ len ≤ 50000 | Pydantic Field validator |
| 邮箱格式 | RFC 5322 | 前端 + 后端双重校验 |
| 密码强度 | 前端最小长度要求 | 前端 validatePassword() |
| URL 合法性 | 后端 requests 验证 | 超时/SSL/连接错误捕获 |
| SQL 注入 | SQLAlchemy ORM 参数化查询 | 天然防护 |
| XSS | 前端 JSX 自动转义 | React 天然防护 |
