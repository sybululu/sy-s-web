### 1.2 核心算法原创性说明

本软件包含以下**具有独创性的核心算法/处理逻辑**，是软著鉴别的重点：

#### 1.2.1 两阶段三重判断合规检测算法（原创）

**位置**: `app.py` 的 `analyze()` 函数 + `binary_predict()` + `roberta_predict()`

**独创性描述**：
本软件设计了一套**两阶段三重判断**的隐私政策违规检测流水线，区别于现有方案的单一模型直接分类：

```
传统方案: 输入句子 → 一个分类模型 → 违规类型
本软件:   输入句子 → 第0关(关键词熔断) → 第1关(二分类模型) → 第2关(LLM二审)
                        ↓                    ↓                    ↓
                     O(1)零推理          RoBERTa二分类        Phi-4 Judge
                     强制违规           高/低/模糊三段式      是/否二分类
```

**三重判断的具体逻辑**：

1. **第 0 关 — 关键词熔断器** (`keyword_circuit_breaker`)
   - 维护 30+ 个流氓表述关键词（如「默认同意」「永久保存」「生物识别」）
   - 命中即判定违规，**无需调用任何 AI 模型**
   - 时间复杂度 O(1)，用于捕获极端明显的违规

2. **第 1 关 — RoBERTa 二分类模型** (`CustomBertBinaryModel`)
   - 基于 `hfl/chinese-roberta-wwm-ext` 微调的二分类模型
   - 将概率空间划分为三个区域：
     - `risk_prob > 0.35`: 高置信违规区 → 直接进入第二阶段
     - `risk_prob < 0.15`: 高置信安全区 → 直接跳过
     - `0.15 ≤ risk_prob ≤ 0.35`: 模糊地带 → 送 Phi-4 二审

3. **第 2 关 — Phi-4 LLM 二审法官** (`llm_judge_violation`)
   - 对二分类不确定的边缘案例，用大语言模型做最终裁决
   - Prompt 设计为只输出「是」或「否」，max_tokens=8 极小化成本
   - 失败时采用保守策略（判定为违规），确保不漏检

**对应代码段**（`app.py` 第 780-965 行）:

```python

    "默认勾选", "默认同意", "视为同意",  # 强制授权类
    "生物识别", "指纹", "人脸识别",         # 敏感数据类
    "永久保存", "无限期",                 # 无期限留存类
    # ... 共 30+ 个关键词

def binary_predict(sentence):             # 第 864 行
    # 第 0 关: 关键词熔断
    matched_kw = keyword_circuit_breaker(sentence)
    if matched_kw:
        return {"is_violation": True, "triggered_by": "keyword", ...}
    
    # 第 1 关: 二分类模型推理
    inputs = tokenizer_roberta(sentence, ...)
    outputs = model_binary(**inputs)
    probs = softmax(outputs.logits)
    
    if risk_prob > 0.35: return {"is_violation": True, ...}   # 高置信违规
    if risk_prob < 0.15: return {"is_violation": False, ...}  # 高置信安全
    
    # 模糊地带: 标记需 LLM 二审
    return {"needs_llm_judge": True, ...}
```

#### 1.2.3 LLM 调用统一稳定性包装器（原创）

**位置**: `app.py` 的 `call_llm_with_retry()` 函数（第 342-437 行）

**独创性描述**：
针对大语言模型 API 调用的不稳定性（超时、限流、过载），设计了**四层容错机制**：

```python
async def call_llm_with_retry(messages, *, max_retries=3, caller="unknown"):
    # Layer 1: Circuit Breaker (熔断器)
    if _is_circuit_open():
        raise LLMCallError("熔断中", retryable=False, error_code="CIRCUIT_OPEN")
    
    for attempt in range(max_retries + 1):
        try:
            # Layer 4: Non-blocking (非阻塞)
            result = await asyncio.to_thread(_llm_call_sync, ...)
            _record_llm_success()
            return result
            
        except LLMCallError as e:
            _record_llm_failure(e)
            
            # Layer 3: Error Classification (错误分类)
            if not e.retryable: raise  # 不可重试立即抛出
            
            # Layer 2: Exponential Backoff (指数退避)
            delay = base_delay * (2 ** attempt)  # 1s → 2s → 4s
            await asyncio.sleep(delay)
```

**四层机制详解**：

| 层级 | 机制 | 作用 | 关键参数 |
|------|------|------|---------|
| L1 | Circuit Breaker | 连续失败 ≥5 次后熔断 60s，避免雪崩 | THRESHOLD=5, COOLDOWN=60s |
| L2 | Exponential Backoff | 可重试错误按指数间隔重试 | base=1.0s, ratio=2^attempt |
| L3 | Error Classification | 区分可重试(429/5xx)和不可重试(401/400) | 10+ 错误码分类 |
| L4 | Non-blocking | asyncio.to_thread 避免阻塞事件循环 | 全部同步调用包装 |

**代码段 C: PolicyModal 阅读进度控制** (`PolicyModal.tsx`)
```typescript
// 独创性: 必须滚动到底部才能点击"同意"按钮
// 包含: 进度条 + 渐变遮罩 + 自动短内容检测
export default function PolicyModal({ isOpen, content, onAccept }) {
  const [scrollProgress, setScrollProgress] = useState(0);
  const [isScrolledToBottom, setIsScrolledToBottom] = useState(false);
  
  const handleScroll = () => {
    const { scrollTop, scrollHeight, clientHeight } = contentRef.current;
    const progress = (scrollTop / (scrollHeight - clientHeight)) * 100;
    setScrollProgress(progress);
    setIsScrolledToBottom(progress > 99);  // 到底部才解锁
  };
  
  // 短内容自动标记为已读
  useEffect(() => {
    if (isOpen && contentRef.current.scrollHeight <= contentRef.current.clientHeight) {
      setIsScrolledToBottom(true);  // 内容不够长，不需要滚动
    }
  }, [isOpen]);
  
  return (
    <>
      {/* 阅读进度条 */}
      <div className="progress-bar">
        <div style={{ width: `${scrollProgress}%` }} />
      </div>
```

#### 1.3.2 后端核心代码段

**代码段 D: 标题行过滤器** (`app.py` 第 663-706 行)
```python
# 独创性: 在送入 RoBERTa 分类前先过滤标题/章节名
# 避免像"一、信息收集"这样的标题被误判为违规内容
_HEADING_PATTERNS = re.compile(r'^('
    r'[一二三四五六七八九十\d]+[\.\、\s]*(?:节|章|条款|部分|)'  # 中文序号
    r'|^\d+[\.\)\s]+'                                          # 数字序号
    r'|(?:信息收集|数据安全|用户权利|Cookie|未成年人|...'       # 标题关键词白名单
    r'|^.{2,20}$'                                               # 短文本(≤20字且无动词)
    r')', re.UNICODE)

_ACTION_KEYWORDS = {'收集', '使用', '共享', '存储', '销毁', ...}  # 行为动词

def is_likely_heading(sentence):
    s = sentence.strip()
    if len(s) > 35: return False           # 过长不是标题
    if _STRONG_HEADING.match(s): return True  # 强标题模式优先
    
    has_action = any(kw in s for kw in _ACTION_KEYWORDS)
    has_subject = any(w in s for ('我们', '将', '会', '可以', '用户', '您'))
    
    if has_subject and has_action: return False  # 有主体+动作 = 正文
    if has_action: return False                   # 有动作 = 正文
    if _HEADING_PATTERNS.match(s): return True     # 弱标题模式
    return False
```

**代码段 E: 加权扣分评分算法** (`app.py` 第 1250-1262 行)
```python
# 独创性: 基于 12 类违规权重的加权扣分制
# 每种违规类型只扣一次（不重复扣分），权重来自论文表 3-1
violation_flags = {info["id"]: 0 for info in INDICATORS.values()}

for violation_id, prob in pred.get("mapped", {}).items():
    dedup_key = (sentence, violation_id)  # (句子, 违规ID) 去重
    if dedup_key not in seen_violations:
        violation_flags[violation_id] = 1  # 每种违规只标记一次
        seen_violations.add(dedup_key)

# 加权求和: penalty = Σ(weight_i × flag_i)
penalty = sum(
    INDICATORS[ID_TO_INDICATOR[v_id]]["weight"] * flag
    for v_id, flag in violation_flags.items()
)
total_score = max(0.0, 100.0 - (penalty * 100.0))  # 百分制

# 风险等级阈值
if total_score >= 70: risk_level = "低风险"
elif total_score >= 40: risk_level = "中等风险"
else: risk_level = "高风险"
```

### 2.3 答辩高频问题准备

**Q1: 这个系统和现有的合规工具有什么区别？**

A: 主要区别有三点：
1. **检测架构创新**：采用两阶段三重判断（关键词熔断 + 二分类 + LLM 二审），而非单一模型分类
2. **RAG 法律检索**：每条违规结论附带真实法律条文依据（FAISS 向量检索 141 条法律），而非通用 LLM 幻构法条
3. **双模整改输出**：同时提供摘要解读（给法务）和合规重写（给产品经理）两种风格

**Q2: AI 模型是自己训练的还是调用的？**

A: 
- **RoBERTa 分类模型**：基于中文法律语料 CAPP-130 数据集自行微调训练，checkpoint 托管在 HuggingFace Hub（`sybululu/bert-moe`）
- **Phi-4 整改模型**：调用 GitHub Models API / HuggingFace Inference API（开源大模型的标准调用方式）
- **嵌入模型**：使用开源的 `paraphrase-multilingual-MiniLM-L12-v2`（Sentence-Transformers）
- **RAG 知识库**：自建的法律条文结构化数据 + FAISS 向量索引

**Q3: 前端的营销站和后台是什么关系？**

A: 采用**单项目双模式架构**——同一个 React 应用根据登录状态渲染不同界面：
- 未登录：展示 Hero 营销站（产品介绍、定价、Demo 演示）
- 已登录：展示 Dashboard 审查工作台（完整的合规审查功能）
- 通过 React Router + AuthContext 实现路由分流

**Q4: 12 类违规指标是怎么来的？**

A: 来源于学术论文研究，覆盖《个人信息保护法》的全部核心义务维度：
- 数据收集类 4 项（I1-I4）：对应第六条最小必要原则
- 数据共享类 3 项（I5-I7）：对应第二十三条共享规则
- 数据留存类 2 项（I8-I9）：对应第十九条和第四十七条
- 用户权利类 3 项（I10-I12）：对应第四十四至五十条
- 权重分配基于各违规类型的法律后果严重程度（总和=1.00）

软著申请表中「技术特点」字段建议从以下角度撰写（选 3-5 条）：

**特点 1: 两阶段三重判断合规检测架构**

**特点 2: RAG 增强型法律依据检索**
> 自建包含 141 条法律条文的领域知识库，基于 Sentence-Transformers 多语言嵌入模型 + FAISS 余弦相似度索引实现语义检索。每条违规结论均附带真实法律条文来源（支持《个人信息保护法》《网络安全法》《数据安全法》GB/T 35273-2020），解决通用 LLM「幻觉法条」的问题。

**特点 3: 11→12 类多标签类别映射算法**
> 针对 RoBERTa 模型 11 类输出与论文 12 类违规指标体系的差距，实现了多标签映射算法（支持 1 个源类别映射到 1-3 个目标违规类型），并通过形式化验证确保 12 类违规指标 100% 全覆盖。

**特点 4: LLM 调用四层容错机制**
> 针对大语言模型 API 的不稳定性，设计了「熔断器(Circuit Breaker) + 指数退避重试(Exponential Backoff) + 错误分类(Error Classification) + 异步非阻塞(asyncio.to_thread)」四层容错架构，在 GitHub Models / HF Inference 三种后端间无缝降级。

**特点 5: 双模式差异化整改 Prompt 工程**
> 为同一违规条款设计「摘要解读」（四段式专业分析，面向法务）和「合规重写」（可直接替换的条款正文，面向产品经理）两种输出模式。其中重写模式内置 12 类违规各自的 EDIT_INSTRUCTIONS 精确操作指令（共 200+ 条编辑规则），配合 3 个 before→after 改写范例驱动输出质量。

**特点 6: 单项目双模式前端架构**
> 同一 React 应用根据认证状态动态切换界面形态：未登录时展示 GSAP+Three.js 驱动的高交互营销站（含 GLSL 波浪背景、3D 翻转卡片、磁吸按钮等 10+ 种动效）；已登录时展示专业的三栏 B 端审查工作台。通过 BrowserRouter + AuthContext 实现零刷新路由分流。
